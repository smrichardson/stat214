Lab 3.2 — LoRA Fine-Tuning Overview

Stat 214 · Spring 2026 · Conceptual guide and pseudocode

The big picture

In 3.1, you generated embeddings using static methods (BoW, Word2Vec, GloVe). In 3.2, you use BERT in two ways:

  1. Pretrained BERT: use the model as-is to extract contextual embeddings. No training involved.
  2. Fine-tuned BERT (LoRA): adapt BERT's representations to the narrative story domain using masked language modeling (MLM), then extract embeddings from the adapted model.

In both cases, once you have per-word embeddings, the rest of the pipeline is identical to 3.1: downsample → delays → ridge → evaluate CC.

Extracting BERT embeddings

BERT takes text and produces a vector per subword token (not per word). The word "playing" might be tokenized into ["play", "##ing"], each getting its own 768-dimensional vector. You need to aggregate subword vectors back to one vector per original word.

Pseudocode: BERT embedding extraction 
function get_word_embeddings(words):
    # words = ["the", "quick", "brown", "fox"]
    
    # 1. Tokenize (tells BERT these are pre-split words)
    subtokens = bert_tokenizer(words, is_split_into_words=True)
    # e.g., ["the", "qu", "##ick", "brown", "fox"]
    
    # 2. Get the word-to-subtoken mapping
    word_ids = subtokens.word_ids()
    # e.g., [0, 1, 1, 2, 3]  — subtokens 1,2 both belong to word 1 ("quick")
    
    # 3. Forward pass through BERT
    hidden_states = bert_model(subtokens)  # shape: (num_subtokens, 768)
    
    # 4. Aggregate: mean-pool subtokens that share the same word_id
    for each unique word_id w:
        word_embedding[w] = mean(hidden_states[k] for k where word_ids[k] == w)
    
    return word_embeddings  # shape: (len(words), 768)
BERT has a maximum sequence length of 512 tokens. Most stories are longer. You need a strategy for handling this: process the story in overlapping windows (e.g., 512 tokens with 50% overlap) and combine results. For tokens appearing in multiple windows, average their representations.

What LoRA does

BERT has ~110M parameters. Full fine-tuning updates all of them, which is slow and risks overfitting on our ~24K words of story data. LoRA freezes all original weights and adds small trainable matrices to specific layers.

The key idea Each attention layer in BERT has weight matrices $W \in \mathbb{R}^{d_1 \times d_2}$ (e.g., $768 \times 768$). Instead of updating $W$ directly, LoRA adds a low-rank update: $$W_{\text{new}} = W_{\text{frozen}} + \frac{\gamma}{r} AB^\top$$ where $A \in \mathbb{R}^{d_1 \times r}$ and $B \in \mathbb{R}^{d_2 \times r}$ are small trainable matrices, $r$ is the rank (e.g., 4 or 8), and $\gamma$ is a scaling factor. Only $A$ and $B$ are trained.

Intuition: the full $768 \times 768$ update has ~590K parameters per layer, but the task-specific adaptation probably only needs a few "directions" of change. A rank-8 factorization captures this with just $768 \times 8 + 768 \times 8 = 12{,}288$ parameters. Think of it like PCA: a few principal components capture most of the variance.

Fine-tuning procedure

The fine-tuning objective is Masked Language Modeling (MLM): the same self-supervised task BERT was originally pretrained with, but now applied to our specific story transcripts.

Pseudocode: LoRA fine-tuning 
# ---- Setup ----
# Load BERT with its MLM prediction head (for training)
mlm_model = load_bert_for_masked_lm("bert-base-uncased")

# Apply LoRA: freeze original weights, add trainable A,B matrices
# to the query and value projections in each attention layer
apply_lora(mlm_model, rank=8, target=["query", "value"])

# ---- Prepare data ----
# Extract story transcripts from raw_text.pkl
stories = load_story_transcripts()

# Tokenize into chunks that fit BERT's 512-token limit
# Use a sliding window with overlap so no text is lost
chunks = []
for story in stories:
    tokens = tokenize(story)
    for start in range(0, len(tokens), stride):
        chunks.append(tokens[start : start + 512])

# ---- Training loop ----
for epoch in range(num_epochs):
    for chunk in shuffle(chunks):
        
        # Mask 15% of tokens (BERT's MLM recipe):
        #   80% → replace with [MASK]
        #   10% → replace with random token
        #   10% → leave unchanged
        masked_chunk, labels = apply_mlm_masking(chunk, prob=0.15)
        
        # Forward pass: predict masked tokens from context
        loss = mlm_model(masked_chunk, labels)
        
        # Backward pass: only LoRA matrices (A, B) get gradients
        loss.backward()
        optimizer.step()

# ---- Save ----
# Save only the LoRA adapter (small file, ~1-5 MB)
save_lora_adapter(mlm_model, "lora_adapter")

Extracting embeddings from the fine-tuned model

Pseudocode: loading and using the fine-tuned model 
# Load the base BERT encoder (NOT the MLM head)
base_encoder = load_bert_model("bert-base-uncased")

# Load the LoRA adapter on top
finetuned_encoder = load_lora_adapter(base_encoder, "lora_adapter")

# Extract embeddings using the same function as before
embeddings = get_word_embeddings(words, finetuned_encoder)
# → shape: (W_s, 768)

# Then feed into the same pipeline as 3.1:
# downsample → delays → ridge → evaluate CC
You train with BertForMaskedLM (which has the prediction head needed for the MLM loss), but you extract embeddings from BertModel (just the encoder). The LoRA adapter modifies the attention layers, which are shared between both model classes. Train with the head, extract without it.

Things to think about for your report

  • Does fine-tuning help? Compare pretrained BERT vs LoRA-finetuned BERT on the same evaluation metrics (mean CC, top 1%/5% CC). If fine-tuning doesn't help, why not? Is there a mismatch between the MLM objective (predicting masked tokens) and the downstream use (extracting pooled embeddings for ridge regression)?
  • Do contextual embeddings predict different voxels? Compute Jaccard similarity between the top-5% voxel sets for BERT vs Word2Vec/GloVe. If BERT predicts well in different brain regions, that suggests it captures different linguistic information.
  • Data leakage: you fine-tune BERT on the story transcripts, then use those same transcripts for ridge regression. Is this leakage? Think carefully about what information the MLM objective uses vs what the ridge regression uses. The MLM task never sees the fMRI data, so the label information (BOLD signal) is not leaked. But the input distribution is shared. Discuss this.
  • Stability: how sensitive are your results to LoRA hyperparameters? If changing the rank from 8 to 16 flips which voxels are "well-predicted," the conclusions are fragile.
